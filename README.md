## What this is

SceneTrip is a persona-aware travel planner. Tell it who's traveling in a
short chat — it assigns one of four personas, builds a profile, and
generates a day-wise itinerary that respects **hard constraints**:

- 90-minute rest blocks for first-time families
- No full-day excursions on a working Tuesday for the bleisure traveler
- Low-walking, accessible, taxi-anchored routes for senior couples
- Hidden-gem-only routing for off-beat explorers

The plan is generated by Gemini using *only real Google Places candidates*
(no hallucinated restaurants), validated against persona rules, and
displayed in an interactive day-card UI. Users can replan with natural
language ("Make it cheaper", "Add vegetarian restaurants") and see the
diff highlighted in place.

**Stack:** React + Vite (frontend), FastAPI + Python (backend),
Gemini 2.5 (LLM), Google Places API New (places), Google Routes API
(travel times). Designed to deploy to Google Cloud Run.

## Demo walkthrough

After running locally (or visiting the Cloud Run URL), the user goes
through six phases in ~90 seconds:

```
intro → onboarding → persona → trip → planning → itinerary
                       ↑                            ↓
                       └────────── replan ──────────┘  (in place)
```

1. **Intro** — single CTA, "Start the conversation"
2. **Onboarding chat** — 13 questions, one at a time, with typing
   indicator and quick-reply chips. The bot lightly acknowledges key
   answers ("Off-Beat energy — got it.")
3. **Persona reveal** — full-card moment showing assigned persona, profile
   attributes, and the **rules engine — applied constraints** list. This
   is the "the app gets me" beat.
4. **Trip questions** — destination, days, budget, stay, restaurants, etc.
5. **Planning loader** — animated step list narrating the actual pipeline
   ("Loading persona rules → Querying Google Places → Estimating Routes →
   Generating itinerary with Gemini → Validating against persona
   constraints").
6. **Itinerary** — expandable day cards with Morning/Afternoon/Evening
   slots, color-coded stop kinds (food/rest/travel/shop/hidden), and a
   **Why this fits you** narrative per day.
   - Below the list: a persistent replan input plus 8 preset chips.
     Submitting either updates the affected day cards in place with diff
     highlights (added stops glow, removed stops strike-through).
   - Right side: a stylized SVG map of Tokyo with an animated route line
     and per-day pins.

### Data flow on a single `/trip/plan` call

1. Client posts `{session_id, trip}`. Server resolves the session and
   pulls the cached `TravelerProfile`.
2. `fetch_candidate_places()` — Google Places (New) Text Search with a
   persona-aware query (`"Tokyo Hidden gems Food local hidden gem
   neighborhood"` for off-beat); persona post-filter drops places with
   too many ratings (tourist-trap heuristic) or wrong accessibility flag.
3. `estimate_route_matrix()` — Google Routes ComputeRouteMatrix in the
   travel mode the persona prefers (WALK / TRANSIT / DRIVE), with a
   haversine fallback when the API isn't reachable.
4. `gemini_client.generate_itinerary()` — builds the **TripSaathi prompt**
   (profile + persona rules + candidate places + route matrix + trip
   details + JSON schema). Forces `response_mime_type=application/json`.
   Parses with a layered JSON extractor (raw → strip code-fences → regex
   fallback).
5. `planner._coerce_day()` — defensive coercion. Normalizes any LLM
   output drift: `kind="cafe"` → `food`, `when="early morning"` →
   `Morning`, malformed stops dropped with a warning. **Never raises.**
6. `validator.validate_itinerary()` — pure function over `(days, rules,
   profile)`. Returns a `ValidationResult` listing rule violations
   (severity warn/error). The API surfaces these but doesn't auto-rewrite
   — the UI can show them, or a future iteration could feed them back into
   Gemini.

## Personas

Defined in [`backend/app/persona_rules.py`](backend/app/persona_rules.py).
These rules drive both the Gemini prompt and the validator — they are the
single source of truth for what each persona means.

| Persona | Max anchors / day | Day window | Walking | Notable guardrails |
|---|---|---|---|---|
| **First-Time Family** | 3 | 10:00 → 20:00 | medium | 90-min post-lunch rest required; food safety; avoid late night |
| **Bleisure Professional** | 2 | 07–09 + 17:30–22 | medium | no full-day excursions on weekdays; reliable Wi-Fi cafés |
| **Off-Beat Explorer** | 2 | 06:00 → 19:00 | high | avoid tourist traps & chain restaurants; lean neighborhood-led |
| **Senior Couple** | 2 | 10:30 → 18:30 | low | avoid steep climbs, queues, loud nightlife; prefer accessible |

Each persona also carries display copy (`display_name`, `tag`, `eyebrow`,
`persona_summary`, `ui_rules`) used by the persona-reveal card.

The frontend has a parallel `PERSONAS` map in
[`frontend/src/data/personas.js`](frontend/src/data/personas.js) for
display only — UI-only fields like `followup` (the question the persona
card subtly asks) and decorative metadata. The hard rules always come
from the backend response.

## Quick start

### Prerequisites

- **Python 3.11 or 3.12** (3.13+ requires `pydantic-core` wheel updates)
- **Node 20+**
- **Optional API keys** — `GEMINI_API_KEY`, `GOOGLE_PLACES_API_KEY`,
  `GOOGLE_ROUTES_API_KEY`. Without them, the app runs in **offline mode**
  using a curated seed itinerary that matches the prototype 1:1.

### Run with Docker Compose (one command)

```bash
cp .env.example .env       # paste your keys, or leave blank for offline mode
docker compose up --build
# → frontend: http://localhost:5173
# → backend:  http://localhost:8080
```

### Run natively

```bash
# Backend
cd backend
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8080
# → docs: http://localhost:8080/docs

# Frontend (in another shell)
cd frontend
npm install
npm run dev
# → http://localhost:5173
```

The Vite dev server proxies `/api/*` to the backend, so the frontend
doesn't need to know the backend URL during local dev.

## Configuration

Copy `.env.example` to `.env` at the repo root. The backend auto-loads it
on startup (via `python-dotenv`).

| Variable | Default | Purpose |
|---|---|---|
| `GEMINI_API_KEY` | *(empty → offline mode)* | Google AI Studio API key. Get one at [ai.google.dev](https://ai.google.dev). |
| `GEMINI_MODEL` | `gemini-2.5-flash-lite` | Model name. `gemini-2.5-flash` is higher quality; `gemini-2.5-flash-lite` is faster + has separate quota bucket. |
| `GOOGLE_PLACES_API_KEY` | *(empty → stub places)* | Places API (New) key. Enables real candidate places. |
| `GOOGLE_ROUTES_API_KEY` | *(empty → haversine fallback)* | Routes API key. Enables real travel-time matrix. |
| `SESSION_TTL_MINUTES` | `120` | In-memory session expiry. |
| `CORS_ORIGINS` | `*` | Comma-separated list. Lock to your frontend origin in prod. |
| `LOG_LEVEL` | `INFO` | Backend log verbosity. |
| `VITE_API_BASE` | `/api` (proxied) | Set this to your Cloud Run URL when building the frontend for production. |

## API reference

Base URL: `http://localhost:8080` (or your Cloud Run URL).
Interactive docs: `GET /docs`.

### `GET /health`

Liveness probe. Used by Docker / Cloud Run health checks.

```json
{ "ok": true, "service": "scenetrip", "version": "0.1.0" }
```

### `POST /profile/create`

Build a `TravelerProfile` from the chat onboarding answers. Permissive by
design — the chat answers map straight in. Only `persona` is required;
everything else gets persona-aware defaults.

**Request**
```json
{
  "name": "Aditya",
  "age": 29,
  "gender": "Prefer not to say",
  "group": "Solo",
  "persona": "off_beat_explorer",
  "pace": "Balanced",
  "budget": "Mid-range",
  "experience": ["Hidden gems", "Culture", "Food"],
  "food": "Vegetarian",
  "walk": "High",
  "access": "None",
  "safety": "Only at night",
  "crowd": "Local hidden gems"
}
```

**Response**
```json
{
  "session_id": "cc302a4489474f15a79744308796aef0",
  "profile": { /* canonical TravelerProfile */ },
  "persona_summary": "You skip the queue and find the alley...",
  "persona_rules": {
    "id": "off_beat_explorer",
    "max_attractions_per_day": 2,
    "ui_rules": ["Max 2 anchored attractions per day", ...],
    "guardrails": { "avoid_tourist_traps": true, ... }
  }
}
```

### `POST /trip/plan`

End-to-end planning pipeline. Resolves the session, fetches places, runs
Routes, prompts Gemini, validates.

**Request**
```json
{
  "session_id": "cc302a4489474f15a79744308796aef0",
  "trip": {
    "destination": "tokyo",
    "days": 5,
    "total_budget": "$2-4k",
    "stay_type": "Boutique stay",
    "area_pref": "City center",
    "include_restaurants": true,
    "include_shopping": true,
    "include_free_time": true
  }
}
```

**Response**
```json
{
  "session_id": "...",
  "used_fallback": false,
  "validation": { "ok": true, "issues": [] },
  "itinerary": [
    {
      "day_number": 1,
      "title": "Yanaka's Old Tokyo Charm",
      "sub": "Explore traditional streets and local art",
      "area": "yanaka",
      "why": "This day focuses on the Yanaka district, known for its traditional atmosphere and local crafts...",
      "walking_minutes": 110,
      "budget_estimate": "¥¥ · ~$60",
      "slots": [
        {
          "when": "Morning",
          "stops": [
            { "name": "Kayaba Coffee", "kind": "food", "meta": "¥ · 7-min walk", "place_id": "..." }
          ]
        },
        { "when": "Afternoon", "stops": [/* ... */] },
        { "when": "Evening",   "stops": [/* ... */] }
      ]
    }
  ]
}
```

`used_fallback: true` means we returned the seed itinerary because the
Gemini call failed (no key, quota, malformed JSON, network) — the UI is
identical either way.

### `POST /trip/replan`

Mutate the current itinerary in place based on a natural-language
instruction.

**Request**
```json
{ "session_id": "...", "instruction": "Add more hidden gems" }
```

**Response**
```json
{
  "session_id": "...",
  "summary": "3 hidden-gem stops added across days 1–3.",
  "changed_days": [0, 1, 2],
  "validation": { "ok": true, "issues": [] },
  "itinerary": [
    /* same shape as /trip/plan, with `_added: true` / `_removed: true`
       flags on the modified stops so the UI can highlight them */
  ]
}
```

## Frontend deep dive

### Phase machine ([App.jsx](frontend/src/App.jsx))

`App` holds five pieces of state and renders one of six phase
components:

```js
{
  phase: 'intro' | 'onboarding' | 'persona' | 'trip' | 'planning' | 'itinerary',
  sessionId: string | null,
  profile: TravelerProfile | null,
  personaRules: object | null,    // received from /profile/create
  destination: string,
  days: ItineraryDay[],
  replanState: { summary, changed[] } | null,
  thinking: boolean,              // replan-in-flight indicator
  planError: string | null,
  planRequestDone: boolean        // gates the planning-loader's last tick
}
```

Phase transitions:

```
setPhase('onboarding') ──▶ ChatWindow(ONBOARDING)
                          │
                          ├─▶ onOnboardingDone(answers)
                          │     └─▶ POST /profile/create
                          │     └─▶ setPhase('persona')
                          │
setPhase('persona')    ──▶ PersonaCard
                          │
                          └─▶ onContinue → setPhase('trip')

setPhase('trip')       ──▶ ChatWindow(TRIP_QS)
                          │
                          └─▶ onTripDone(answers)
                                ├─▶ setPhase('planning')
                                └─▶ POST /trip/plan (background)

setPhase('planning')   ──▶ Planning(actuallyDone)
                          │  ticks animated steps; when planRequestDone
                          │  flips to true, jumps to last step then
                          │  setPhase('itinerary')

setPhase('itinerary')  ──▶ ItineraryCard + replan input
                          │
                          └─▶ onReplan(instruction)
                                └─▶ POST /trip/replan
```

### Components

- **`ChatWindow.jsx`** — conversational state machine. One question at a
  time, typing indicator before each bot turn, light "ack" lines after
  key answers (name, persona, destination). Accepts a `script` prop so
  it's reused for both onboarding and trip questions.
- **`AnswerPanel.jsx`** — renders the input shape based on question
  `kind`: `text`, `chip`, `chip-rich` (label + sub), or `chip-multi`.
  Every question also offers a "Use suggested · {default}" skip.
- **`PersonaCard.jsx`** — full-card persona reveal. Display copy from
  the local `PERSONAS` map, but the *rules list* comes from the backend
  response so it always matches what the engine actually enforces.
- **`ItineraryCard.jsx`** — day-card list with collapsible bodies, color-
  coded stop dots, persona-fit reason ribbon, and the persistent replan
  input at the bottom.
- **`MapPane.jsx`** — pure SVG map of Tokyo. Animated route polyline,
  per-day pins with pop-and-pulse animations, neighborhood labels that
  fade in. Other destinations show a placeholder.
- **`Planning.jsx`** — ticks through the pipeline steps independently
  of the network request, then jumps to the last step when the API
  resolves. Keeps the loader honest — the steps map 1:1 to what the
  backend is actually doing.
- **`Intro.jsx`**, **`TopBar.jsx`**, **`Bubble.jsx`** — straightforward.

### Styling

[`frontend/src/styles/scenetrip.css`](frontend/src/styles/scenetrip.css)
is the prototype's CSS, ported verbatim. Color tokens:

## Backend deep dive

### `schemas.py`

Pydantic 2 models. Strict (`extra="forbid"`) where it matters, permissive
where the LLM might surprise us. Notable models:

- **`TravelerProfile`** — strict. Built from chat answers in `main.py`.
- **`Stop`** — strict `kind` enum. The defensive coercion in
  `planner._coerce_day` filters LLM output before construction so this
  never throws on real traffic.
- **`SessionState`** — the in-memory record: `session_id` → profile +
  trip + current itinerary + last replan summary.
- **`ValidationResult`** — `ok` + list of `ValidationIssue` (each with
  `day_number`, `rule`, `severity`, `message`).

### `persona_rules.py`

A dict-of-dicts. Every persona has the same shape so the planner and
validator can iterate generically. Adding a 5th persona = drop in a new
key with the same structure; no other code changes needed.

### `google_places.py`

Async wrapper around Places API (New) Text Search. Builds a query from
destination + experience types + persona-specific keywords. Uses
`X-Goog-FieldMask` to keep responses small. Post-filter drops:

- `user_ratings_total > 5000` for `off_beat_explorer` (tourist trap
  heuristic)
- `wheelchair_accessible == False` for `senior_couple`
- `amusement_park` / `zoo` for `bleisure_professional`

If `GOOGLE_PLACES_API_KEY` is unset, returns a hand-curated stub list of
Tokyo / Lisbon places that match the prototype's seed itineraries — the
demo never breaks for missing keys.

### `google_routes.py`

Async wrapper around Routes API ComputeRouteMatrix. Persona drives the
travel mode (`WALK` / `TRANSIT` / `DRIVE` for senior taxi-anchored). When
no key is set or coords are missing, falls back to a haversine + speed-
of-mode heuristic. Flags legs that exceed the persona's per-leg walking
cap (12 min for senior, 60 min for off-beat).

### `gemini_client.py`

See [Gemini prompt architecture](#gemini-prompt-architecture-tripsaathi)
below.

### `planner.py`

Three responsibilities:

1. **`plan_trip()`** — orchestrates Places → Routes → Gemini → coerce →
   validate. Always returns `(days, validation, used_fallback)`.
2. **`_coerce_day()`** — defensive Pydantic adapter. Normalizes free-form
   LLM output (`kind: "cafe"` → `"food"`, `when: "early morning"` →
   `"Morning"`), drops malformed stops with a warning instead of raising.
   This is why the API never crashes on a Gemini response.
3. **`_apply_replan_rules()`** — the deterministic replan engine. See
   [Replan engine](#replan-engine).

### `main.py`

FastAPI app. Loads `.env` from the repo root. Adds permissive CORS
middleware (configurable). Maintains an in-memory session dict with TTL
GC on every read. Sessions self-expire after `SESSION_TTL_MINUTES`.

## Validator

[`backend/app/validator.py`](backend/app/validator.py) is a **pure
function** — given an itinerary, persona rules, and a profile, returns a
`ValidationResult`. No I/O, no mutation.

The 7 checks it currently runs:

| # | Check | Severity | Triggered for |
|---|---|---|---|
| 1 | `max_attractions_per_day` | error | All personas — anchored stops (kind ≠ food/rest/travel) |
| 2 | `mandatory_rest_after_lunch` | warn | First-Time Family — needs a `kind: "rest"` in the Afternoon slot |
| 3 | `avoid_loud_nightlife_areas` | error | Senior Couple — keyword scan for nightclub/club/dive |
| 4 | `senior_evening_density` | warn | Senior Couple — Evening slot has >1 stop |
| 5 | `avoid_chain_restaurants` | warn | Off-Beat Explorer — keyword scan |
| 6 | `no_full_day_excursion_weekday` | error | Bleisure — keyword scan |
| 7 | `food_pref_alignment` | warn | Veg/vegan/jain/halal travelers — food stops must mention the preference |

The validator never auto-rewrites. The API returns the issues in the
response. With a paid Gemini setup, you can implement a re-prompt loop
(send the issues back to Gemini and ask for a corrected itinerary) — the
validator interface is ready for it.

## Gemini prompt architecture (TripSaathi)

[`backend/app/gemini_client.py`](backend/app/gemini_client.py) builds a
single structured prompt:

```
SYSTEM INSTRUCTIONS
  You are TripSaathi, SceneTrip's planning engine. ...
  - Respect every persona constraint as a hard rule, not a suggestion.
  - Use ONLY the candidate places provided. Do not invent place names.
  - Output strict JSON matching the schema. No markdown. No preamble.
  - Each day must include a `why` field.

TRAVELER PROFILE: { full TravelerProfile JSON }
PERSONA RULES:    { rules_for(persona) JSON — hard constraints }
CANDIDATE PLACES: [ compact list of PlaceResult objects from Places API ]
ROUTE MATRIX:     [ sequential legs with mode, duration_min, distance_m ]
TRIP DETAILS:     { destination, days, budget, stay_type, ... }

OUTPUT JSON SCHEMA:
  { itinerary: [
      { day_number, title, sub, area, why,
        slots: [ { when: Morning|Afternoon|Evening,
                   stops: [ { name, kind, meta, place_id, ... } ] } ],
        walking_minutes, budget_estimate
      } ] }

Now produce {N} days. Output ONLY JSON.
```

Configured with:
```python
genai.GenerativeModel(
  GEMINI_MODEL,
  generation_config={
    "response_mime_type": "application/json",   # forces structured output
    "temperature": 0.6,
  },
)
```

JSON parsing is layered:
1. Try `json.loads(text)` directly.
2. If text starts with `\`\`\``, strip the code-fence.
3. Else, regex out the first `{...}` block.
4. If all fail → fall back to seed itinerary, log the error.

The model is invoked via `asyncio.to_thread()` because the official SDK
is synchronous. Cold-start cost is amortized — `google.generativeai` is
imported lazily inside the function.

## Replan engine

The replan endpoint is **deterministic by design** — it mirrors the
prototype's `applyReplan()` exactly, so demo behavior is consistent
whether or not Gemini is reachable.

[`planner._apply_replan_rules()`](backend/app/planner.py) recognizes 8
intent patterns from the user's instruction:

| Intent keywords | Effect |
|---|---|
| `more relaxed`, `relax` | Add 90-min rest in the named day's afternoon; remove last evening stop. |
| `hidden`, `off-beat`, `off beat` | Add 3 hidden-gem afternoon stops to days 1–3. |
| `reduce walking`, `less walking` | Insert taxi transfers in every morning; trim day 3. |
| `cheap`, `budget` | Mark all `¥¥¥` stops as removed; add a convenience-store lunch on day 2. |
| `vegetarian`, `vegan` | Add T's Tantan (vegan ramen) to every evening. |
| `crowd`, `avoid` | Mark famous-name stops (Shibuya Scramble, Takeshita, Sky Tree) as removed. |
| `shop` | Add Kuramae makers + Daikanyama T-Site to days 3 and 4. |
| `evening` + `free` | Mark all evening stops removed; add an "Open block" rest on each. |

Anything else: a generic "adjusted per: {instruction}" hidden stop is
added to day 1 (or the day mentioned in the text via `day N` regex).

Modified stops are tagged with `_added: true` or `_removed: true` so the
frontend can render them with diff styling (terracotta glow / strike-
through fade) without recomputing what changed.

To upgrade this to a true LLM-driven replan, swap the function body to
call `gemini_client.replan_itinerary(current_days, instruction, profile,
rules)` — the validator runs the same way on the result.

## Troubleshooting

### `used_fallback: true` even though I set `GEMINI_API_KEY`

Three common causes:

1. **Wrong model name.** `gemini-2.0-flash-exp` was deprecated. Use
   `gemini-2.5-flash` or `gemini-2.5-flash-lite`.
2. **Free-tier quota exhausted.** 20 requests/day per model. The server
   logs will show `429 ResourceExhausted ... Quota exceeded`. Switch
   models (separate quota buckets) or enable billing.
3. **`.env` not loaded.** `main.py` calls `load_dotenv()` from the repo
   root. If you renamed your env file, point at it explicitly or export
   the vars in your shell.

Check the backend logs — the wrapper logs the underlying error explicitly:
```
WARNING gemini_client · Gemini quota exhausted, falling back to seed itinerary: ...
```

### Pydantic `literal_error` on `Stop.kind`

This shouldn't happen anymore — `planner._coerce_day` normalizes any
weird `kind` from the LLM. If you see it, the LLM returned a structure
where `kind` lives somewhere unexpected (e.g. nested deeper). File an
issue with the response body.

### Frontend can't reach the backend in dev

Vite's proxy (in [`vite.config.js`](frontend/vite.config.js)) sends
`/api/*` to `http://localhost:8080`. If your backend is on a different
port, set `VITE_API_TARGET=http://localhost:NEW_PORT npm run dev`.

### `pydantic-core` build failure on `pip install`

Means you're on Python 3.13+ and the wheel doesn't exist yet. Either
downgrade to Python 3.11/3.12, or upgrade the `pydantic` pin in
`requirements.txt` to the latest (≥2.10) which has 3.13 wheels.

### Session "expired" mid-demo

`SESSION_TTL_MINUTES=120` by default. For a longer demo, bump it in
`.env` and restart the backend.

## Roadmap

Out of MVP scope, in priority order:

1. **More destinations.** The Tokyo seed + Places filter is generic; add
   curated stubs and map polygons for Lisbon, Mexico City, Kyoto.
2. **LLM-driven replan.** Replace the deterministic replan engine with a
   second Gemini call. Validator stays the same.
3. **Re-prompt loop.** When the validator returns errors, send them back
   to Gemini and ask for a corrected day. Caps at 2 retries.
4. **Auth + persistence.** Replace the in-memory session with Postgres-
   backed accounts so users can save and revisit plans.
5. **Booking handoff.** Deep-links to OpenTable / Klook / Booking.com
   for stops that have hooks. No actual booking server-side.
6. **Multi-city trips.** Tokyo → Kyoto → Osaka in one session. Adds a
   travel-mode segment between cities.
7. **Real-time signals.** Weather adjustment, crowd density, opening
   hours overrides.
